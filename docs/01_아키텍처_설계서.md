# 멀티테넌트 RAG 챗봇 아키텍처 설계서

> **대상:** 개발팀
> **목적:** 본 문서를 읽고 각 컴포넌트의 역할·경계·구현 방향을 이해하여 작업을 시작할 수 있도록 한다.
> **상태:** 설계 확정 단계 (일부 항목은 `결정 필요사항` 참고)

> ⚠️ **구현 현황(as-built):** 이 문서는 설계 의도다. 실제 구현은 [12_구현_아키텍처.md](./12_구현_아키텍처.md) 참고. 주요 차이:
> - **MCP 서버(FAQ·일반)는 업체별 외부 독립 프로젝트**(FastAPI+FastMCP)로 분리. 오케스트레이터는 원격 MCP 클라이언트.
> - **§6 A안의 "전용 벡터DB 인스턴스"는 컬렉션 분리로 확정**(인스턴스 분리는 운영 전환). 데이터 적재는 벤더 서버 소유.
> - 라우팅 기본 규칙 기반, verify OFF, 임베딩 백엔드 선택식(`hash` 기본 | `fastembed` 실제 모델), LLM 로컬 Claude CLI 옵션.
> - FAQ 는 0단계 즉답(`match_faq`) **+ 검색 소스(`search_faq` top-K)** 두 역할 — 복합 질문은 0단계를 건너뛰고 rag/agent 가 FAQ·문서·도구를 병합(정공법).

## 1. 한눈에 보기 (TL;DR)

- **무엇을 만드나:** 업체(테넌트)별로 격리된, 검색 기반(RAG) 응답 챗봇.
- **핵심 전략:** 비싼 처리를 최대한 안 하도록 **계단형 필터** 구조.
  요청을 `FAQ 즉답 → 도구 기반 응답 → 복잡 작업` 순으로 점점 비싼 단계에 넘긴다.
- **3가지 설계 축**
  1. **계단형 필터링** — 싸고 빠른 단계에서 최대한 걸러낸다.
  2. **MCP = 도구·데이터의 경계** — 도메인/업체별로 서버를 분리한다.
  3. **벡터 검색 = 선택의 두뇌** — "어떤 FAQ인가", "어떤 도구인가"를 의미 기반으로 고른다.
- **멀티테넌시:** 법적 격리 + 비용 귀속을 위해 **업체별 FAQ MCP 서버 분리(A안)**. 단 **코드는 단일 템플릿**, 데이터·인스턴스만 격리.

## 3. 핵심 설계 원칙

### 원칙 1. 계단형 필터링 — "싸고 빠른 것부터 거른다"

요청을 단계별로 거르며, 각 단계는 앞 단계보다 비싸고 느리다. 앞에서 처리되면 뒤는 실행되지 않는다.

| 순서 | 단계 | 비용 | 처리 대상 | LLM |
|---|---|---|---|---|
| 0 | FAQ 시맨틱 인터셉트 | 💸 거의 0 | 정형화된 자주 묻는 질문 | ❌ |
| 1 | 도구 벡터검색 + 오케스트레이터 | 💸 중 | 일반 검색/단순 작업 | ✅ 1회 |
| 2 | 다단계 도구 호출(루프) | 💸 고 | 복잡·다단계 작업 | ✅ 다회 |

> **의도:** 트래픽의 다수를 차지하는 단순/정형 질문을 0~1단계에서 처리하여 비용·지연을 최소화하고, 복잡한 질문만 비싼 경로로 보낸다.

### 원칙 2. MCP = 도구·데이터의 경계 (LLM 아님)

- **MCP 서버는 LLM이 아니다.** 도구(함수)를 표준 인터페이스로 노출하는 "창구"일 뿐이다.
- LLM은 **오케스트레이터** 한 곳에 있다. MCP 서버에는 LLM이 들어가지 않는다.
- MCP로 분리하는 이유: **모듈화·독립 배포·권한 경계·재사용·네임스페이스(이름 충돌 해소)**.

```
오케스트레이터 LLM (도구를 "선택"하는 두뇌)  ← LLM ✅
   └─ MCP 서버 (도구를 "제공"하는 창구)        ← LLM ❌
```

### 원칙 3. 벡터 검색 = 선택의 두뇌

기존 RAG가 "질문 → 관련 **문서** 검색"이라면, 본 구조는 같은 메커니즘을 두 군데 더 재활용한다.

| 대상 | 검색하는 것 | 사용 위치 |
|---|---|---|
| 문서 | 관련 문서 청크 | 도메인 RAG 도구 내부 |
| **FAQ** | 의미가 같은 FAQ 질문 | 0단계 시맨틱 인터셉트 |
| **도구** | 질문에 맞는 도구(함수) | 1단계 Tool RAG |

> **이점:** 강의에서 다룬 RAG 파이프라인(임베딩 → 벡터검색 → Hybrid → Rerank)을 **그대로 재활용**한다. 추가 학습/인프라 부담이 적다.

## 4. 컴포넌트별 명세

### 4.1 대화 관리 레이어

| 항목 | 내용 |
|---|---|
| 역할 | 대화 히스토리 유지 + 후속 질문을 독립 질문으로 재작성 |
| 왜 필요 | "그거 더 알려줘" 같은 후속 질문은 그 자체로 검색 불가 → 맥락 결합 필요 |
| 단기 메모리 | 세션 단위 대화 히스토리 |
| 장기 메모리 | (선택) 사용자별 선호/과거 대화 — `트렌드 영상 실습 코드/long-term memory(Memori)` 참고 |
| LLM | ✅ 질문 재작성에 가벼운 모델 사용 |

### 4.2 테넌트 라우터 (결정적)

| 항목 | 내용 |
|---|---|
| 역할 | `company_id` → 해당 업체 FAQ MCP 서버 주소로 라우팅 |
| 방식 | **결정적** (LLM/벡터검색 불필요). A사 유저는 무조건 A사 서버 |
| 구현 | **서버 레지스트리(매핑 테이블) 조회** — 하드코딩 금지 |
| company_id 출처 | **고객용: 요청에서 직접 수신(비회원)** / 관리자용: 인증 토큰. 라우터가 추측하지 않음 |

```yaml
# 서버 레지스트리 예시 (설정/DB로 관리, 업체 추가 시 한 줄 추가)
tenants:
  A사: { faq_server: "https://a-faq.internal", vector_db: "qdrant-a:6333" }
  B사: { faq_server: "https://b-faq.internal", vector_db: "qdrant-b:6333" }
  C사: { faq_server: "https://c-faq.internal", vector_db: "qdrant-c:6333" }
```

> ⚠️ **하드코딩(if-else) 금지.** 업체 추가가 코드 수정·재배포를 유발하면 운영이 무너진다. 매핑은 외부 설정/DB에서 조회한다.

### 4.3 FAQ MCP 서버 (0단계 · 시맨틱 인터셉트)

| 항목 | 내용 |
|---|---|
| 역할 | 정형화된 질문을 **의미 기반**으로 매칭해 즉답 (LLM·생성 단계 없음) |
| 매칭 | 키워드 ❌ → **벡터 유사도** ✅ ("몇 시까지 해요?" ≈ "영업시간이 어떻게 되나요?") |
| 흐름 | 질문 임베딩 → 벡터검색 → (Rerank) → 유사도 ≥ 임계값이면 저장된 답 반환, 아니면 통과 |
| 임계값 | **보수적으로 높게** 설정. 확실할 때만 즉답, 애매하면 통과(놓쳐도 1단계가 답함) |
| 격리 | 업체별 **독립 벡터DB 인스턴스** 연결 (법적 격리) |
| 배치 | 오케스트레이터 **앞단의 독립 관문** (도구로 노출하지 않음 — 그래야 즉답 이점 유지) |

**임계값 운영 가이드**

| 임계값 | 효과 | 리스크 |
|---|---|---|
| 높게(예: 0.90) | 확실할 때만 즉답 | 일부 FAQ 놓침(→ 1단계가 처리, 안전) |
| 낮게(예: 0.75) | 더 많이 즉답 | **오답 즉답 위험** (다른 질문에 FAQ 답) |

> **권장:** 높게 시작 → 로그 모니터링하며 조정. 애매 구간(0.80~0.85)만 가벼운 LLM으로 "같은 뜻?" 검증하는 옵션 고려.

**진화(선택):** 자주 들어오는 질문 + 좋은 답변을 자동으로 FAQ DB에 적재 → 시맨틱 캐시화. 시간이 갈수록 0단계 처리율 상승.

### 4.4 도구 벡터 검색기 (1단계 · Tool RAG)

| 항목 | 내용 |
|---|---|
| 역할 | 전 도메인 MCP 서버의 도구를 의미 기반으로 검색해 관련 Top-K만 LLM에 노출 |
| 왜 필요 | 도구가 많으면(50개+) LLM 선택 정확도 급락 + 토큰/비용 폭증 |
| 인덱싱 | 각 도구의 `description`을 임베딩 → 벡터DB. 메타데이터에 `{server, name, description}` |
| 검색 | Hybrid(이름=BM25 + 용도=벡터) → Rerank로 Top-K 정제 |
| 주의 | **검색 누락 = 치명적** (필요한 도구가 후보에서 빠지면 LLM이 시도조차 못함) → Top-K 넉넉히 + Rerank |
| 핵심 | 도구 `description` 품질이 검색 품질을 좌우 → 설명을 검색 잘 되게 작성 |

### 4.5 오케스트레이터 LLM (1~2단계)

| 항목 | 내용 |
|---|---|
| 역할 | 추려진 Top-K 도구 중 선택·호출, 결과로 답변 생성 |
| 단일 LLM | 도메인별 LLM을 따로 두지 않음 (Tool RAG로 도구를 좁히므로 LLM 1개로 충분) |
| 다단계 | 한 번 검색으로 안 끝나는 작업은 **루프**: `현재상태 → 도구 재검색 → 선택·호출 → 결과 반영` 반복 |
| 구현 | LangGraph (ToolNode + 조건부 루프) |

### 4.6 도메인 MCP 서버들

| 항목 | 내용 |
|---|---|
| 역할 | 도메인별 실제 도구 실행 (문서검색/요약, DB조회, 결제 등) |
| LLM | ❌ 없음 (도구 제공만) |
| RAG 도구 | 문서 도메인 서버의 `search_documents`는 내부에서 Hybrid 검색 + Rerank 사용 |
| 분리 기준 | 도메인 단위(코드/배포/권한 경계) |

### 4.7 LLM 모델 분리 전략 (비용·속도 최적화)

단계마다 같은 모델을 쓰지 않는다.

| 작업 | 모델 성격 | 예시 |
|---|---|---|
| 질문 재작성, 라우팅 판단, 관련성 평가 | 가볍고 빠름 | Claude Haiku 4.5 |
| 최종 답변 생성, 복잡한 추론 | 고품질 | Claude Sonnet 4.6 / Opus 4.8 |

> 보조 단계를 가벼운 모델로 처리하면 멀티스텝 구조의 지연·비용을 크게 줄일 수 있다.

### 4.8 자동완성 / 입력 유도 (입력 시점)

> 🆕 **신규 구현 항목.** 강의의 Multiquery Retriever(CH05.06, `generate_queries`)와 **다른 기능**이다.
> Multiquery는 "사용자 질문을 실시간으로 변형해 검색 recall을 높이는 것"이고, 본 기능은 "문서에서 미리 생성한 질문 후보를 입력 시점에 추천하는 것"이다. 혼동 금지.

| 항목 | 내용 |
|---|---|
| 역할 | 사용자가 질문을 타이핑할 때, **답할 수 있는 질문**을 추천해 올바른 입력으로 유도 |
| 효과 | FAQ 적중률↑(0단계 즉답↑) → 비용↓ / 모호한 질문↓ / 빈손 응답↓ |
| 핵심 속성 | 추천 후보 = **반드시 답이 있는 질문** → 검색 실패가 구조적으로 거의 없음 |
| 위치 | 계단형 필터보다 **앞**, 즉 **사용자 입력 시점** |
| 제약 | 매 키스트로크 호출 → **초고속(수십 ms) 필수**, LLM 실시간 호출 ❌ |

**후보 풀 3대 소스**

| 소스 | 강점 | 생성 시점 |
|---|---|---|
| **문서 기반 생성** ⭐ | 커버리지 최대, 콜드스타트 해결 | 오프라인(인덱싱 시) |
| FAQ | 정확히 답 준비된 질문(즉답 연결) | 상시 |
| 질문 로그 | 실제 사용자 표현·인기도 | 운영하며 축적 |

**문서 기반 후보 생성 (오프라인, 핵심)**

문서는 서술문이므로 **질문으로 변환**해야 한다. 인덱싱 단계에서 LLM으로 청크당 "답할 수 있는 질문" N개 생성 → 자동완성 풀 + 질문 임베딩 DB에 적재. (실시간 아님 → 비용 부담 없음)

```
문서원문: "영업시간은 평일 9시~18시입니다."
   → LLM 변환 → "영업시간이 어떻게 되나요?" / "몇 시까지 영업하나요?" / "주말에도 하나요?"
   → 자동완성 풀에 적재 (company 태그 포함)
```

**구현 방식 (조합 권장)**

| 방식 | 시점 | 비용 |
|---|---|---|
| ① prefix 매칭 (주력) | 매 타이핑 | 매우 저 |
| ② 시맨틱 추천 (보강) | 입력 멈춤(디바운스 150~300ms) | 저 |
| ③ LLM 후속질문 제안 | 답변 후/턴 종료 시에만 | 중(가끔) |

**prefix 저장소 선택 — ES 불필요**

후보가 **미리 만든 유한 풀**이라 무거운 ES 없이도 prefix 매칭이 된다. 특히 멀티테넌트 A안에선 업체별 ES 클러스터가 부담이라 **ES 미사용이 운영상 유리**.

| 저장소 | 적합 상황 |
|---|---|
| **인메모리 Trie** ⭐ | 후보 유한·중간 규모, 매우 빠름 |
| DB 인덱스(prefix/pg_trgm) | 이미 RDB 사용 중, 추가 인프라 0 |
| Redis(RediSearch) | 빠른 캐시 필요 |
| Elasticsearch | 후보 방대 + 오타교정/퍼지/복잡 랭킹 필요 시에만 |

> 초기엔 **Trie/DB로 시작 → 오타교정·대규모·복잡 랭킹이 필요해지면 ES 승격**.

**멀티테넌트 주의:** 자동완성 후보도 **company_id 필터 필수**. A사 유저에게 B사 질문이 추천되면 격리 위반. 문서 기반 후보도 **업체별로 분리 생성·격리**한다.

## 5. 데이터 흐름: 인덱싱(오프라인) vs 응답(온라인)

RAG는 항상 두 프로세스로 나뉜다. **파싱·청킹·임베딩은 응답 시점이 아니라 사전 작업이다.**

### 인덱싱 (사전 1회 / 데이터 추가 시)

```
원본(문서 / FAQ / 도구설명)
   → 파싱(Extract)        [문서만]
   → 청킹(Chunking)       [문서만]
   → 임베딩(Embedding)
   → 벡터DB 적재(Load)
        └─→ 🆕 LLM 질문 생성(문서 청크 → 답 가능한 질문 N개)
                → 자동완성 풀(Trie/DB) + 질문 임베딩 DB 적재 [업체별 격리]
```

- **문서 RAG DB:** 도메인 문서 → 파싱·청킹·임베딩
- **FAQ DB(업체별):** 업체 담당자 제출 FAQ → 임베딩 (답변은 메타데이터로 저장)
- **도구 DB:** 각 도구의 description → 임베딩
- **자동완성 풀(업체별):** 문서 청크 → LLM 질문 생성 → prefix 저장소 + 질문 임베딩 (4.8 참고)

### 응답 (요청마다 / 파싱·청킹 없음)

```
질문 → (재작성) → 임베딩 → 벡터검색 → (Rerank) → [즉답 | 도구선택→생성]
```

| 구분 | 인덱싱 | 응답 |
|---|---|---|
| 시점 | 사전 / 데이터 추가 시 | 요청마다 |
| 파싱·청킹 | 있음(문서) | 없음 |
| 임베딩 대상 | 문서/FAQ/도구 | **질문** |

## 6. 멀티테넌시 설계 (A안: 업체별 서버 분리)

**결정:** 법적 격리 + 비용 귀속(업체별 분리 과금/정산)을 위해 **업체별 FAQ MCP 서버를 분리**한다.

### 핵심 원칙: "서버 N개 ≠ 코드 N벌"

```
        📦 FAQ MCP 서버 템플릿 (코드 1벌)
                    │  ← 같은 코드, config만 다름
   ┌────────────────┼────────────────┐
   ▼                ▼                ▼
 A사 인스턴스        B사 인스턴스        C사 인스턴스
 company=A         company=B         company=C
 vector_db=A       vector_db=B       vector_db=C
 threshold=0.85    threshold=0.88    threshold=0.85
```

- **코드는 단일 템플릿.** 업체 구분은 환경변수/설정으로만.
- 로직 변경 = 템플릿 수정 후 전체 재배포(CI/CD).
- 업체 추가 = 새 설정으로 인스턴스 1개 + 레지스트리 한 줄 추가 (코드 작성 X).

### 격리 수준

| 수준 | 격리 | 법적 요구 충족 |
|---|---|---|
| 공유 DB + 컬렉션 분리 | 논리적 | 보통 부족 |
| **DB 인스턴스 분리** ⭐ | 물리적(데이터) | 대체로 충족 |
| 완전 독립 스택 | 완전 | 최고 |

> 법적 격리가 이유이므로 **최소 "업체별 독립 벡터DB 인스턴스"**. 즉 A사 MCP 서버는 A사 전용 벡터DB만 연결한다. (서버 안에 DB를 "내장"이 아니라, 서버가 전용 DB를 "연결".)

### 두 가지 라우팅을 혼동하지 말 것

| 무엇을 고르나 | 방법 | 이유 |
|---|---|---|
| **어느 업체 서버?** | 결정적(레지스트리 조회) | A사 유저는 무조건 A사. 벡터검색 ❌ (오격리·법적 위반 위험) |
| **그 안에서 어느 FAQ?** | 의미기반 벡터검색 | 표현 달라도 같은 뜻 매칭 |

### 운영 자동화 (A안 필수)

| 장치 | 목적 |
|---|---|
| 템플릿 + IaC/CI/CD | 인스턴스 찍어내기·일괄 업데이트 |
| 공통 FAQ 업로드 파이프라인 | 담당자 제출(시트 등) → 자동 임베딩·적재 |
| 중앙 모니터링/로깅 | 서버 N개를 업체 태그로 단일 대시보드 |
| 서버 레지스트리 | 테넌트→서버/DB 매핑 (라우터 참조) |
| 비용 태깅 | 업체별 리소스 태그 → 분리 정산 |
| (선택) 스케일-투-제로 | idle 업체 서버 비용 절감(서버리스/오토스케일) |

> ⚠️ **FAQ 업로드 파이프라인 표준화는 필수.** 수작업 적재는 업체 증가 시 운영을 마비시킨다.

## 11. 용어집

| 용어 | 의미 |
|---|---|
| RAG | 검색 증강 생성. 외부 지식을 검색해 LLM 답변에 활용 |
| MCP 서버 | 도구를 표준 인터페이스로 노출하는 서버. **LLM 아님** |
| 시맨틱 인터셉트 | LLM 전에 의미 기반으로 정형 질문을 가로채 즉답 |
| Tool RAG | 도구(함수) 선택을 벡터 검색으로 수행 |
| Hybrid 검색 | 벡터(의미) + 키워드(BM25) 결합 검색 |
| Reranker | 검색 후보의 순위를 재정렬해 정밀도 향상 |
| 테넌트 | 업체(고객사) 단위. 멀티테넌시 = 다수 업체 격리 운영 |
| 결정적 라우팅 | LLM/검색 없이 식별값(company_id)으로 경로 확정 |
| 임계값(Threshold) | FAQ 즉답 여부를 가르는 유사도 기준점 |
| 자동완성(입력 유도) | 타이핑 시 답 가능한 질문을 추천해 입력 품질을 높이는 기능 |
| Trie(접두어 트리) | prefix 매칭에 특화된 경량 자료구조. 자동완성 저장소로 적합 |
| Multiquery Retriever | 사용자 질문을 실시간 변형해 검색 recall을 높이는 기법(CH05.06). 자동완성과 다름 |

## 부록 A. "왜 이렇게 설계했나" 핵심 의사결정 요약

| 결정 | 대안 | 채택 이유 |
|---|---|---|
| FAQ를 인터셉트로 (도구 ❌) | 오케스트레이터 도구로 노출 | 즉답·저비용 이점 유지 (LLM 앞단 처리) |
| FAQ 매칭 = 벡터검색 | 키워드 매칭 | 표현이 달라도 같은 의미 매칭 |
| 도구 선택 = Tool RAG | 모든 도구를 LLM에 노출 | 도구 과다 시 선택 정확도·토큰 문제 해결 |
| 오케스트레이터 LLM 1개 | 도메인별 LLM(계층 Agent) | Tool RAG로 충분, 비용·지연 절감 |
| 업체별 서버 분리(A안) | 단일 서버 + 메타필터(B안) | 법적 격리 + 비용 귀속 |
| 코드 단일 템플릿 | 업체별 개별 코드 | A안의 운영 부담(코드 중복) 제거 |
| 업체 선택 = 결정적 라우팅 | 벡터검색으로 업체 선택 | 오격리·법적 위반 방지 |
| 자동완성 후보 = 문서기반 생성 | 키워드 추출만 | 답 가능한 질문만 추천 → 검색 실패 구조적 최소화 |
| 자동완성 저장소 = Trie/DB | Elasticsearch | 유한 풀 + 멀티테넌트라 ES는 과함(필요 시 승격) |
```
