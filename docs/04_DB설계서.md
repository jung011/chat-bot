# DB 설계서

> **대상:** 백엔드 개발팀
> **목적:** 챗봇이 사용하는 데이터 저장소(벡터DB / 관계형DB / 캐시)의 구조와 멀티테넌트 격리 전략을 정의한다.
> **관련 문서:** [01_아키텍처_설계서.md](./01_아키텍처_설계서.md) · [02_백엔드_패키지구조.md](./02_백엔드_패키지구조.md) · [03_API명세서.md](./03_API명세서.md)
> **전제:** DB는 **기존 로컬 도커 컨테이너에 연결**한다(신규 기동 없음). 연결정보는 `.env`/`configs`로 외부화.
> **상태:** 초안 (v1)

> ⚠️ **구현 현황(as-built):** [12_구현_아키텍처.md](./12_구현_아키텍처.md) §5 참고. 주요 차이:
> - **FAQ 격리 = 컬렉션 분리**(`faq_<id>`), **인스턴스 분리 아님**(공유 Qdrant). documents/tools/autocomplete_q 는 단일 컬렉션 + `company_id` 필터.
> - 공유 컬렉션엔 **`company_id` KEYWORD payload 인덱스**(검색 가속). `tools` 는 `company_id` 태깅(업체별 도구).
> - 적재는 벤더 MCP 서버가 소유(임베딩 포함). 테넌트 레지스트리 SSOT = `configs/tenants.yaml`(`companies` 테이블 아님).
> - **임베딩 백엔드 선택식**(`EMBEDDING_BACKEND`): `hash`(기본) | `fastembed`(실제 모델). 공유 컬렉션은 오케스트레이터·벤더 동일 모델 필수(§08 §9.3).
> - **⭐업체 백엔드 자체 DB 추가(§6 데이터 성격별 분리)**: 정형/실시간 데이터(메뉴·매장·배달·주문)는 업체 백엔드의 **별도 DB**(예: `pizza_shop`, PostgreSQL)가 소유. 오케스트레이터/MCP 는 이 DB 에 직접 접근하지 않고 **백엔드 API 경유**(§01 §2.2).

## 1. 데이터 저장소 개요

| 저장소 | 용도 | 멀티테넌트 처리 |
|---|---|---|
| **벡터 DB (Qdrant)** | 의미 검색 (FAQ / 문서 / 도구 / 자동완성 질문) | FAQ는 **업체별 인스턴스 격리(A안)**, 그 외는 컬렉션+필터 |
| **관계형 DB (PostgreSQL)** | 세션·메시지·테넌트·피드백·로그 등 메타데이터 | `company_id` 컬럼 + 행 단위 필터 |
| **캐시 (Redis)** | 단기 대화 캐시, 자동완성 prefix, 시맨틱 캐시, 레이트리밋 | key prefix에 `company_id` 포함 |
| **⭐업체 백엔드 DB (벤더 소유)** | 정형/실시간 서비스 데이터(메뉴·가격·매장·재고·주문) | 업체별 **별도 DB**(예: `pizza_shop`). 오케스트레이터 미접근 — 백엔드 API 경유 |

> 위 3종(Qdrant/Postgres/Redis)은 **오케스트레이터가 직접 쓰는** 저장소다. **업체 백엔드 DB** 는 외부 벤더 백엔드(예: `pizza-backend`, FastAPI)가 소유하며, 같은 Postgres 인스턴스라도 **오케스트레이터 메타DB(`app`)와 다른 데이터베이스(`pizza_shop`)** 로 분리한다. 데이터 성격별 분리 원칙: **정형·실시간 → 백엔드 DB(API 경유)**, **비정형 텍스트 → 벡터DB(RAG)**, **정형 Q/A → FAQ 컬렉션**. (§01 §2.2, §08 §9, §12 §4.2)

## 2. 멀티테넌트 격리 전략 (핵심)

§01 §6의 A안(법적 격리 + 비용 귀속)에 따라 저장소별로 격리 수준을 달리한다.

| 데이터 | 격리 수준 | 근거 |
|---|---|---|
| **FAQ (업체별)** | **독립 벡터DB 인스턴스** | 법적 격리 + 비용 귀속(A안). A사 서버는 A사 DB만 연결 |
| 문서 RAG | 컬렉션 분리 + `company_id` 필터 | 중앙 운영, 논리 격리 |
| 도구(Tool RAG) | 단일 컬렉션(공용 도구) | 테넌트 무관 공용 |
| 자동완성 질문 | 컬렉션/풀 분리 + `company_id` 필터 | 문서 기반 생성물, 업체별 격리 |
| 관계형 데이터 | 단일 DB + `company_id` 컬럼 필터 | 메타데이터, 행 단위 격리 |

> ⚠️ **모든 검색/조회에 `company_id` 필터를 강제**한다. 애플리케이션 레이어에서 누락 시 격리 위반 → 공통 의존성(`api/deps.py`)에서 테넌트 컨텍스트를 주입하고, 레포지토리 계층에서 필터를 항상 적용.

## 3. 벡터 DB (Qdrant)

### 3.1 컬렉션 목록

| 컬렉션 | 위치 | 벡터 | 용도 |
|---|---|---|---|
| `faq` | **업체별 인스턴스** | 질문 임베딩 | FAQ 시맨틱 인터셉트(0단계) |
| `documents` | 중앙 인스턴스 | 청크 임베딩 | 문서 RAG |
| `tools` | 중앙 인스턴스 | 도구 description 임베딩 | Tool RAG |
| `autocomplete_q` | 중앙(또는 업체별) | 생성 질문 임베딩 | 자동완성 시맨틱 보강 |

### 3.2 페이로드 스키마 (메타데이터)

**`faq` 컬렉션** (업체별 인스턴스 → company_id는 인스턴스로 이미 격리되나 일관성 위해 보관)
```jsonc
{
  "id": "faq_a_001",
  "vector": [/* 질문 임베딩 */],
  "payload": {
    "company_id": "A사",
    "question": "영업시간이 어떻게 되나요?",
    "answer": "평일 9시~18시입니다.",      // 즉답에 사용
    "category": "운영",
    "source": "manual | auto_cache",       // 담당자 등록 / 자동 캐시
    "updated_at": "2026-06-17T00:00:00Z"
  }
}
```

**`documents` 컬렉션**
```jsonc
{
  "id": "doc_chunk_123",
  "vector": [/* 청크 임베딩 */],
  "payload": {
    "company_id": "A사",                   // ★ 검색 시 필수 필터
    "doc_id": "policy_2026",
    "chunk_index": 5,
    "text": "영업시간은 평일 9시...",
    "title": "운영 정책",
    "source_uri": "s3://.../policy.pdf",
    "page": 3
  }
}
```

**`tools` 컬렉션**
```jsonc
{
  "id": "tool_search_documents",
  "vector": [/* description 임베딩 */],
  "payload": {
    "server": "documents",                 // 소속 MCP 서버
    "name": "search_documents",
    "description": "사내 문서를 검색합니다",
    "params_schema": { /* JSON Schema */ }
  }
}
```

**`autocomplete_q` 컬렉션**
```jsonc
{
  "id": "acq_001",
  "vector": [/* 생성 질문 임베딩 */],
  "payload": {
    "company_id": "A사",                   // ★ 필터
    "question": "주말에도 영업하나요?",
    "origin_doc_id": "policy_2026",        // 출처 문서
    "popularity": 0                        // 로그 기반 가중치
  }
}
```

### 3.3 검색 파라미터 규약

| 컬렉션 | 검색 방식 | 필터 | 비고 |
|---|---|---|---|
| `faq` | 벡터 + 임계값 | (인스턴스 격리) | 임계값 이상만 즉답 |
| `documents` | Hybrid(벡터+BM25) + Rerank | `company_id` | §01 §4.4 |
| `tools` | Hybrid + Rerank | (공용) | Top-K 추출 |
| `autocomplete_q` | 벡터(보강용) | `company_id` | prefix는 Redis/Trie 우선 |

> 임계값(FAQ)·Top-K(Tool RAG)·Rerank 적용 여부는 설정값으로 외부화한다.

## 4. 관계형 DB (PostgreSQL)

### 4.1 ERD 개요

```
companies (테넌트)
   │ 1
   ├─< admins               (업체 담당자 계정 — 고객은 비회원이라 없음)
   ├─< sessions ─< messages ─< message_feedback   (비회원, anon_id 선택)
   ├─< faq_sources          (담당자 제출 원본/적재 이력)
   ├─< query_logs           (질문 로그 → 자동완성/분석)
   └─< index_jobs           (인덱싱/적재 작업 상태)
```

### 4.2 테이블 명세

#### `companies` — 테넌트
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `company_id` | varchar PK | 테넌트 식별자 |
| `name` | varchar | 업체명 |
| `faq_server_url` | varchar | 업체 FAQ MCP 서버 주소(레지스트리) |
| `faq_vector_db` | varchar | 업체 FAQ 벡터DB 접속정보 |
| `faq_threshold` | float | FAQ 즉답 임계값(업체별) |
| `status` | varchar | active/suspended |
| `created_at` | timestamptz | |

> 테넌트 레지스트리(§01 §4.2)를 DB로 관리하는 형태. `configs/tenants.yaml`과 둘 중 하나로 단일화(결정 필요).

#### `admins` (구 `users`)
고객은 **비회원이라 계정 없음.** 이 테이블은 **업체 담당자(관리자) 계정만** 보관한다.
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `admin_id` | varchar PK | |
| `company_id` | varchar FK | 소속 테넌트 |
| `role` | varchar | admin (운영자 등 확장 가능) |
| `created_at` | timestamptz | |

#### `sessions` — 대화 세션 (비회원)
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `session_id` | varchar PK | 추측 어려운 난수 ID |
| `company_id` | varchar | ★ 요청에서 수신한 테넌트(필터/혼입 방지) |
| `anon_id` | varchar NULL | 익명 클라이언트/디바이스 식별(선택, 분석용) |
| `title` | varchar | |
| `created_at` / `updated_at` | timestamptz | |

> 비회원이라 `user_id`(계정) 대신 익명 `anon_id`(선택). 세션 접근은 `session_id`+`company_id`로 조회.

#### `messages` — 메시지
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `message_id` | varchar PK | |
| `session_id` | varchar FK | |
| `company_id` | varchar | 비정규화(필터 성능) |
| `role` | varchar | user / assistant |
| `content` | text | 메시지 본문 |
| `route` | varchar | faq_intercept / rag / agent (분석용) |
| `sources` | jsonb | 근거(FAQ/문서/도구) |
| `usage` | jsonb | 토큰 사용량 |
| `created_at` | timestamptz | |

#### `message_feedback` — 피드백
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | bigserial PK | |
| `message_id` | varchar FK | |
| `company_id` | varchar | |
| `rating` | varchar | up / down |
| `reason` | text | |
| `created_at` | timestamptz | |

#### `faq_sources` — FAQ 원본/적재 이력
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | bigserial PK | |
| `company_id` | varchar FK | |
| `question` | text | |
| `answer` | text | |
| `status` | varchar | pending/embedded/failed |
| `vector_id` | varchar | Qdrant 포인트 id 매핑 |
| `created_at` | timestamptz | |

#### `query_logs` — 질문 로그
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | bigserial PK | |
| `company_id` | varchar | |
| `anon_id` | varchar NULL | 익명 식별(선택) |
| `raw_query` | text | 원문 질문 |
| `route` | varchar | 처리 경로 |
| `matched` | bool | FAQ/검색 적중 여부 |
| `created_at` | timestamptz | |

> 자동완성 `log` 소스 및 인기도(popularity) 산출, 미응답 질문 분석에 사용.

#### `index_jobs` — 인덱싱 작업
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `job_id` | varchar PK | |
| `company_id` | varchar | |
| `type` | varchar | faq / documents |
| `scope` | varchar | full / incremental |
| `status` | varchar | queued/running/done/failed |
| `stats` | jsonb | 처리 건수 등 |
| `created_at` / `finished_at` | timestamptz | |

### 4.3 인덱스 권장

| 테이블 | 인덱스 |
|---|---|
| `sessions` | `(company_id, updated_at desc)`, `(company_id, anon_id)` |
| `messages` | `(session_id, created_at)`, `(company_id, route)` |
| `query_logs` | `(company_id, created_at)`, `(company_id, matched)` |
| `faq_sources` | `(company_id, status)` |

## 5. 캐시 (Redis)

| 용도 | Key 패턴 | TTL | 비고 |
|---|---|---|---|
| 단기 대화(세션 컨텍스트) | `chat:{company_id}:{session_id}` | 수 시간 | 빠른 히스토리 조회 |
| 자동완성 prefix | `ac:{company_id}` (Trie/Sorted Set) | 갱신 시 | §01 §4.8, prefix 매칭 |
| 시맨틱 캐시 | `sc:{company_id}:{hash}` | 설정 | 동일/유사 질문 답 재사용 |
| 레이트리밋 | `rl:{ip}` 또는 `rl:{company_id}:{anon_id}` | 윈도우 | 비회원 → IP/세션 기준 |

> 모든 key에 `company_id` 포함 → 캐시 레벨에서도 테넌트 분리.

## 7. 결정 필요사항

- [ ] 현재 로컬 도커에 떠 있는 DB 목록 확정 (Qdrant/Postgres/Redis 외 추가 여부)
- [ ] 테넌트 레지스트리 단일 소스: `companies` 테이블 vs `configs/tenants.yaml`
- [ ] FAQ 벡터DB **인스턴스 격리** 구체화(업체별 Qdrant 인스턴스 vs 별도 방식) — 법무 확인
- [ ] `messages.content` 보관 정책(보존기간/개인정보 마스킹)
- [ ] `autocomplete_q` 위치: 중앙 컬렉션+필터 vs 업체별 컬렉션
- [ ] ORM/마이그레이션 도구(SQLAlchemy + Alembic 등) 선정
