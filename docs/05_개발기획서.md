## 1. 배경 & 문제

외식 업체는 고객의 반복 문의(영업시간, 메뉴, 배달지역, 주문방법 등)에 사람이 일일이 응대한다. 문의의 다수가 **정형화된 질문**이라, 자동 응대로 운영 부담을 줄이고 응답을 빠르게 할 수 있다.

- 같은 질문이 다양한 표현으로 들어온다("몇 시까지 해요?" = "영업시간?").
- 업체마다 메뉴·정책·답변이 **완전히 다르다** → 업체별 격리 필요.

## 2. 목표 (Goals)

| 구분 | 목표 |
|---|---|
| 사용자 가치 | 고객이 메뉴·영업·배달·주문 문의에 **즉시·정확한** 답을 받는다 |
| 운영 가치 | 반복 문의 자동화로 업체 응대 부담↓ |
| 기술 가치 | 강의 RAG 스택을 재활용한 **멀티테넌트 RAG 챗봇** 레퍼런스 구축 |

### 성공 지표 (KPI)

| 지표 | 목표(파일럿) |
|---|---|
| FAQ 즉답 적중률 | 핵심 문의의 70%+ 를 0단계(FAQ)에서 처리 |
| 평균 응답 시작 시간(TTFT) | 스트리밍 첫 토큰 1.5초 이내 |
| 미응답("모르겠어요") 비율 | 10% 이하 |
| 답변 정확도(피드백 up 비율) | 85%+ |
| 건당 LLM 비용 | 목표 단가 이내(계단형 필터로 절감) |

## 3. 대상 사용자 (Tenants & Users)

**파일럿 테넌트 3개 업체**

| company_id | 업체 | 도메인 특성 |
|---|---|---|
| `pizza` | 피자집 | 피자/사이드 메뉴, 토핑 커스텀, 세트 |
| `chinese` | 중국집 | 면/밥/요리, 점심특선, 단체주문 |
| `chicken` | 치킨집 | 후라이드/양념, 부분육, 세트·음료 |

**사용자 유형**

| 유형 | 설명 |
|---|---|
| 고객(end user) | 각 업체 챗봇에 문의하는 손님 |
| 업체 담당자(admin) | FAQ 등록·문서 제공·운영 모니터링 |

## 4. 범위 (Scope)

### In Scope (파일럿)
- 업체별 **FAQ 시맨틱 인터셉트**(영업시간/배달지역/결제수단 등 정형 문의 즉답)
- **문서 RAG** 기반 메뉴·정책 문의 응답
- **자동완성**(입력 유도, 문서기반 질문 생성)
- 멀티턴 대화(질문 재작성, 세션 히스토리)
- 업체별 격리(서버/DB/페르소나)
- 관리자: FAQ 업로드, 인덱싱 트리거

### Out of Scope (파일럿 제외, 향후)
- 실제 **결제·주문 트랜잭션 처리**(외부 POS/주문 시스템 연동) — 안내까지만
- 음성/전화 채널
- 다국어(한국어 우선)
- 장기 메모리(개인화) — 후순위
- Dockerfile/배포 자동화 — 추후(기존 로컬 도커 사용)

## 5. 핵심 시나리오 (예시)

| # | 사용자 | 질문 | 처리 경로 |
|---|---|---|---|
| 1 | 피자집 고객 | "몇 시까지 배달돼요?" | FAQ 인터셉트 → 즉답 |
| 2 | 중국집 고객 | "점심특선에 뭐뭐 있어요?" | 문서 RAG(메뉴) → 생성 |
| 3 | 치킨집 고객 | "반반 시키면 양념 안 매운맛 돼요?" | 문서 RAG(메뉴/옵션) → 생성 |
| 4 | 고객 | "ㅇ" 입력 중 | 자동완성 → "영업시간 알려주세요" 추천 |
| 5 | 피자집 담당자 | FAQ 30건 업로드 | 관리자 API → 임베딩 적재 |

> ⚠️ 업체 격리: 피자집 고객에게 중국집 메뉴/FAQ가 절대 노출되면 안 됨.

## 7. 리스크 & 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| FAQ 오답 즉답(임계값 낮음) | 신뢰도↓ | 임계값 보수적, 애매구간 통과 |
| 업체 데이터 혼입 | **법적/신뢰 치명** | 모든 검색에 company_id/인스턴스 격리 강제 |
| 메뉴/정책 변경 미반영 | 잘못된 안내 | 재인덱싱 정책·갱신 주기 운영 |
| LLM 비용 초과 | 운영비↑ | 계단형 필터 + 모델 티어 분리 |
| 개인정보(주소/전화) 노출 | 법적 | 로깅 마스킹·보존정책(§10) |
