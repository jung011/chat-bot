# LLM / 프롬프트 설계서

> **대상:** 백엔드/AI 개발팀
> **목적:** 오케스트레이션 각 단계에서 어떤 모델로, 어떤 프롬프트로 판단·생성하는지 정의한다.
> **관련 문서:** [01_아키텍처_설계서.md](./01_아키텍처_설계서.md) · [02_백엔드_패키지구조.md](./02_백엔드_패키지구조.md) (app/orchestration, app/llm)
> **상태:** 초안 (v1)

> ⚠️ **구현 현황(as-built):** [12_구현_아키텍처.md](./12_구현_아키텍처.md) 참고. 주요 차이:
> - **LLM 백엔드 2종**: `LLM_PROVIDER=anthropic`(API) | `claude_cli`(로컬 Claude Code CLI, 키 불필요).
> - **라우팅 기본 = 규칙 기반**(`ROUTE_MODE=rule`) — LLM 호출 0. RAG 의 LLM 호출을 1회(generate)로 줄임(속도).
> - **agent 단계는 Sonnet 사용**(설계의 Opus 아님 — Opus 상수는 정의돼 있으나 미매핑).
> - **verify 노드 기본 OFF**(`VERIFY_ENABLED=false`) — 경량 판사 오기각 방지. 생성 프롬프트가 컨텍스트 외 답변 금지를 강제.
> - 답변 실패 유형(A/C)은 mode="fail" 로만 표기(별도 분기 미구현).

## 1. 모델 티어 전략

단계별로 모델을 분리해 속도·비용·품질을 균형 맞춘다(§01 §4.7).

| 용도 | 모델 | 이유 |
|---|---|---|
| 질문 재작성, 라우팅, 관련성/환각 평가 | **Claude Haiku 4.5** | 가볍고 빠른 판단 |
| 최종 답변 생성 | **Claude Sonnet 4.6** | 품질/비용 균형(기본) |
| 복잡한 다단계 추론(Agent) | **Claude Opus 4.8** | 어려운 케이스만 |

> 모델 ID는 `app/llm/models.py`에서 상수로 관리하고, 단계별 매핑을 설정으로 둔다.

## 2. 공통 프롬프트 원칙

- **역할·제약·출력형식**을 시스템 프롬프트에 명시한다.
- **근거(컨텍스트)에 없는 내용은 답하지 않는다**(환각 억제).
- 업체 **페르소나/톤**을 시스템 프롬프트에 주입한다(업체별).
- 판단 노드는 **JSON 등 구조화 출력**으로 강제해 파싱 안정성을 확보한다.
- 사용자 입력은 데이터로 취급(프롬프트 인젝션 방지, §10 연계).

## 3. 업체 페르소나 (시스템 프롬프트 변수)

업체별 톤/제약을 설정으로 주입. `{persona}` 변수로 합성.

| company_id | 페르소나 예시 |
|---|---|
| `pizza` | "피자집 친절한 직원. 토핑·세트 추천에 능숙. 밝고 캐주얼한 말투." |
| `chinese` | "중국집 직원. 점심특선·단체주문 안내에 능숙. 정중하고 간결." |
| `chicken` | "치킨집 직원. 후라이드/양념·부분육 옵션 안내. 친근한 말투." |

> 페르소나·영업정보는 업체 설정(`configs`/`companies`)에서 로드하여 주입한다.

## 4. 노드별 프롬프트 명세

### 4.1 질문 재작성 (rewrite) — Haiku

대화 히스토리를 반영해 후속 질문을 독립 질문으로 변환.

**System**
```
너는 사용자 질문을 검색 가능한 독립 질문으로 재작성한다.
- 대화 맥락의 지시어("그거","아까 그거")를 구체 명사로 치환한다.
- 의미를 바꾸지 말 것. 새 정보를 추가하지 말 것.
- 이미 독립적이면 원문을 그대로 반환한다.
출력: 재작성된 질문 한 줄만.
```
**User**
```
[대화 히스토리]
{history}
[현재 질문]
{question}
```

### 4.2 라우팅 (route) — Haiku

질문을 처리 경로로 분류(계단형 필터의 1단계 분기).

**System**
```
사용자 질문을 아래 중 하나로 분류한다.
- "chitchat": 인사/잡담 등 검색 불필요
- "rag": 메뉴/정책/매장 정보 등 검색으로 답할 수 있음
- "agent": 여러 단계/도구 조합이 필요한 복잡한 요청
출력(JSON): {"route":"chitchat|rag|agent","reason":"간단한 근거"}
```
**User** `{question}`

> 0단계 FAQ 인터셉트는 LLM 이전에 처리되므로 라우터 분류 대상이 아니다.

### 4.3 도구 선택 (tool_select) — Sonnet/Haiku

Tool RAG로 추려진 후보 도구 중 선택(또는 직접 답변).

**System**
```
아래는 이 질문에 관련된 도구 후보다. 필요한 도구를 선택해 호출하라.
- 도구가 불필요하면 호출하지 말고 바로 답하라.
- 후보에 적합한 도구가 없으면 모른다고 답하라.
[도구 후보]
{tool_candidates}   # name, description, params_schema
```
**User** `{question}`

### 4.4 답변 생성 (generate) — Sonnet

검색 컨텍스트 기반 최종 답변. 업체 페르소나 적용.

**System**
```
{persona}
아래 [컨텍스트]만 근거로 답한다. 컨텍스트에 없으면 "확인이 어렵다"고 안내하고 추측하지 않는다.
- 가격/시간/지역 등 수치는 컨텍스트 값을 정확히 사용한다.
- 간결하고 친절하게. 필요한 경우 항목으로 정리한다.
- 답변 마지막에 근거 출처가 있으면 자연스럽게 덧붙인다.
[컨텍스트]
{retrieved_context}
[업체 영업정보]
{business_info}
```
**User**
```
[대화 히스토리]
{history}
[질문]
{question}
```

### 4.5 검증 (verify) — Haiku (선택)

생성 답변의 근거 충실성·관련성 점검.

**System**
```
답변이 [컨텍스트]에 근거하는지 평가하라.
출력(JSON): {"grounded":true|false,"issues":["근거 없는 주장 ..."]}
```
**User**
```
[컨텍스트]{retrieved_context}
[답변]{answer}
```
> `grounded=false`면 재생성 또는 "확인 어려움" 안내로 폴백.

## 5. 가드레일

| 항목 | 처리 |
|---|---|
| 범위 밖 질문(정치/의료 등) | 정중히 거절, 업체 관련 안내로 유도 |
| 프롬프트 인젝션("지시 무시해") | 사용자 입력을 데이터로만 취급, 시스템 지시 우선 |
| 개인정보 요청/노출 | 수집·노출 금지(§10) |
| 주문·결제 확정 요청 | 파일럿 범위 밖 → 안내만, 실제 처리 거절 |
| 환각 위험 | 컨텍스트 외 답변 금지 + verify 노드 |

## 6. 출력/스트리밍 규약

- 최종 생성은 **토큰 스트리밍**(§03 SSE `token`).
- 근거는 `sources` 이벤트로 별도 전송.
- 판단 노드(JSON)는 스트리밍하지 않고 내부 처리.

### 6.1 표준 폴백 문구 (사용자 노출)

| 상황 | 문구 |
|---|---|
| LLM 타임아웃(60초 초과) | "답변을 준비하는 데 시간이 조금 오래 걸리고 있어요. 😅 잠시 후 다시 시도해 주시거나, 질문을 조금만 더 구체적으로 입력해 주시면 더 빠르게 도와드릴게요." |
| 타임아웃(짧은 버전) | "지금 답변이 지연되고 있어요. 잠시 후 다시 시도해 주세요. 🙏" |
| 근거 없음/확인 어려움 | "죄송해요, 해당 내용은 정확히 확인이 어려워요. 매장에 직접 문의해 주시면 정확히 안내받으실 수 있어요." |
| 범위 밖 질문 | "그 부분은 도와드리기 어려워요. 메뉴·영업시간·배달 관련해서는 무엇이든 물어봐 주세요!" |

> 타임아웃 정책(60초)·상태코드는 [03_API명세서.md](./03_API명세서.md) §1.5.1 참조.

## 7. 프롬프트 운영

- 프롬프트는 **버전 관리**(템플릿 파일 + 버전 태그).
- 변경 시 평가셋으로 회귀 검증(§11 평가 문서 연계).
- 프롬프트 변수: `{persona}`,`{history}`,`{question}`,`{retrieved_context}`,`{business_info}`,`{tool_candidates}`.

## 8. 대화 이력 / 컨텍스트 관리

멀티턴이 길어지면 토큰 한도를 초과하고 비용·지연이 증가한다. 컨텍스트를 관리한다.

| 항목 | 정책 |
|---|---|
| 토큰 예산 | 모델별 입력 한도 내에서 system + 컨텍스트 + 히스토리 합산 관리 |
| 히스토리 절단 | 최근 N턴 유지, 초과분은 절단 |
| 히스토리 요약 | 절단 대신 과거 대화를 요약해 압축(선택) |
| 검색 컨텍스트 | Rerank Top-N만 투입(§07), 과다 주입 금지 |
| 재작성 활용 | 질문 재작성으로 장황한 히스토리 의존도↓ |

> 단기 히스토리는 Redis(`chat:{company_id}:{session_id}`, §04 §5)에서 조회. 장기 메모리는 파일럿 범위 밖(§05).

## 9. 병렬 / 다단계 도구 호출

- 모델이 한 응답에 **여러 도구를 동시에 요청**할 수 있다 → 오케스트레이터는 tool_call들을 순회 실행하고 결과를 각각 반영한다.
- 서로 의존 없는 도구는 **병렬 실행** 가능(지연 단축). 의존 있는 작업은 순차.
- 도구 결과를 반영해 모델이 **또 도구를 부르면 반복**(LangGraph 루프, `max_iters` 상한으로 무한루프 방지).
- 멱등(읽기전용) 도구만 재시도. 비멱등 도구는 재시도 금지(§08 §6).

## 10. 답변 실패 처리 (타임아웃과 별개)

> ⚠️ **타임아웃과 혼동 금지.** 타임아웃(§03 §1.5.1, `504`)은 "시간 안에 못 끝낸 것"이라 **재시도가 유효**하다. 여기서 다루는 "답변 실패"는 **정상적으로 끝났지만(200) 답을 못 찾은 것**이라 **재시도가 무의미**하다. 둘은 상태코드·메시지·재시도 여부가 다르다.

### 10.1 실패 유형 구분

| 유형 | 원인 | 상태 | 재시도 |
|---|---|---|---|
| A. 적합한 도구 없음 | Tool RAG 후보에 도구 없음(미구현 기능) | 200 | ❌ |
| B. 데이터 없음 | 도구 실행됐으나 `fail()`(정상 부정응답) | 200 | ❌ |
| C. 근거 무관/빈손 | RAG 검색 결과가 질문과 무관 | 200 | ❌ |
| D. 범위 밖 질문 | 잡담/정치/의료 등 | 200 | ❌ |
| (참고) 타임아웃 | 시간 초과(미완) | 504 | ✅ |
| (참고) 시스템 오류 | LLM/DB 장애 | 503 | ✅ |

> A·C가 "도구/근거가 없어 답을 못 찾은" 실패. B는 사유를 그대로 전달하면 됨. D는 가드레일(§5).
> **A·B·C·D는 에러가 아니라 200 정상 응답**으로 폴백 메시지를 반환한다.

### 10.2 처리 3단계 (A·C 기준)

```
도구/근거 없음
  → ① 솔직히 인정      (지어내지 않음 — 환각 금지)
  → ② 대안 제시        (아래 우선순위)
  → ③ 실패 기록        (query_logs matched=false → 개선 루프)
```

**② 대안 우선순위**
1. **답할 수 있는 관련 질문 추천** — 자동완성 질문 풀(§07) 재활용. "혹시 이런 게 궁금하셨나요?"
2. **매장 연결** — `get_store_info`의 연락처 안내(사람 에스컬레이션).
3. **가능 범위 안내** — "메뉴·영업시간·배달 관련은 도와드릴 수 있어요."

> ❌ 답변 실패에는 **"잠시 후 다시 시도해 주세요"를 쓰지 않는다**(재시도해도 같은 결과 → 거짓 희망). 재시도 유도는 타임아웃 전용.

### 10.3 메시지 (유형별)

| 유형 | 문구 |
|---|---|
| A·C (추천 가능) | "죄송해요, 그 부분은 정확히 확인이 어려워요. 🙏 혹시 이런 게 궁금하셨을까요? {추천질문 2~3개} — 또는 매장에 직접 문의해 주시면 정확히 안내받으실 수 있어요(☎ {매장번호})." |
| A·C (추천 없음) | "죄송해요, 해당 내용은 제가 도와드리기 어려워요. 메뉴·영업시간·배달 관련은 무엇이든 물어봐 주세요!" |
| B (데이터 없음) | 도구 `fail()` 사유를 자연스럽게 전달 (예: "입력하신 주소는 현재 배달 가능 지역 정보가 없어요.") |
| D (범위 밖) | §6.1 범위 밖 문구 |

### 10.4 개선 루프 (장기)

- 실패 = **콘텐츠/도구의 구멍 신호**. `query_logs.matched=false`(§04 §4.2)로 적재.
- 운영자가 미응답 질문 주기 분석 →
  - 빈도 높음 → **FAQ 추가**(담당자 요청) 또는 **문서 보강**(재인덱싱, §07 §5)
  - 기능 필요(A) → **새 도구 구현**(§08 §5.5)
- 유형 분리 집계: **A 비율↑ = 도구 추가 필요**, **C 비율↑ = 콘텐츠 보강 필요**.

### 10.5 연속 실패 (선택)

- 한 세션에서 **2~3회 연속 실패** 시 → 즉시 매장 연결 우선 제안(이탈 방지).

## 11. 결정 필요사항
- [ ] 단계별 모델 매핑 최종 확정(특히 generate: Sonnet vs Opus 기준)
- [ ] verify 노드 상시 적용 vs 조건부(비용/지연 트레이드오프)
- [ ] 페르소나 텍스트 업체 담당자 검수 여부
- [ ] 거절/폴백 문구 표준화
- [ ] 히스토리 관리 방식(절단 vs 요약) 및 유지 턴 수
- [ ] 도구 호출 루프 `max_iters` 상한값
- [ ] 답변 실패 시 관련 질문 추천 개수, 연속 실패 임계(에스컬레이션) 기준
