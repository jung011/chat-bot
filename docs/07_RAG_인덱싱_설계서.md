# RAG / 인덱싱 파이프라인 설계서

> **대상:** 백엔드/AI 개발팀
> **목적:** 문서·FAQ·도구를 검색 가능하게 만드는 인덱싱(오프라인) 및 검색(온라인) 전략을 정의한다.

## 1. 두 프로세스 (재확인)

| 구분 | 인덱싱(오프라인) | 검색(온라인) |
|---|---|---|
| 시점 | 사전/데이터 추가 시 | 요청마다 |
| 파싱·청킹 | 있음(문서) | 없음 |
| 임베딩 대상 | 문서/FAQ/도구/질문 | 질문 |

## 2. 데이터 소스 (업체별)

| 소스 | 예시(피자집/중국집/치킨집) | 적재 대상 |
|---|---|---|
| 메뉴판 | 메뉴/가격/옵션/세트 | `documents` |
| 매장 정보 | 영업시간/위치/주차/연락처 | `documents` + FAQ |
| 배달 정책 | 배달지역/최소주문/배달비/시간 | `documents` + FAQ |
| 정형 FAQ | 담당자 제출 Q&A | `faq`(업체 인스턴스) |

> 모든 소스는 **업체별로 분리 적재**. 피자집 데이터는 피자집 저장소에만.

## 3. 인덱싱 파이프라인

```
원본(메뉴판/정책 문서)
  → 파싱(Extract)        # PDF/HWP/이미지 → 텍스트/표
  → 청킹(Chunking)       # 의미 단위 분할
  → 임베딩(Embedding)
  → 벡터DB 적재(Load: documents, company_id 태그)
       └─→ 질문 생성(LLM) → autocomplete_q + 자동완성 풀 적재
```

### 3.1 파싱(Extract)
- 강의 스택 활용(Part3 CH02): Docling / Upstage Document Parse / PyMuPDF4LLM 등.
- 메뉴판은 **표·가격 구조 보존**이 중요 → 표 인식 우선.
- 산출: 텍스트 + 표(마크다운) + 메타(페이지/제목).

### 3.2 청킹(Chunking)
- 기본: 의미 기반 청킹(Part3 CH03). 메뉴 항목/정책 단락 단위.
- 메뉴는 **항목 단위로 끊어** "메뉴명+가격+옵션"이 한 청크에 들어가게.
- 메타데이터 태깅(`title`,`category`,`page`).

### 3.3 임베딩(Embedding)
- 텍스트: 일반 임베딩(Part3 CH04). 인덱싱/질의에 **동일 모델** 사용.
- (멀티모달 메뉴 이미지 필요 시 ColPali/Voyage 고려 — 파일럿 후순위.)

### 3.4 적재(Load)
- Qdrant `documents` 컬렉션, payload에 `company_id` 필수(§04 §3.2).

### 3.5 질문 생성(자동완성 풀) — 핵심
- 각 청크에서 LLM으로 "답 가능한 질문" N개 생성(§01 §4.8).
- 산출 질문 → `autocomplete_q` 임베딩 + prefix 저장소(Trie/Redis).
- 후처리: 중복 제거, 너무 일반적/모호한 질문 필터.
```
청크: "마르게리따 피자 L 25,000원, 토핑 추가 가능"
 → "마르게리따 피자 가격이 얼마예요?"
 → "피자 토핑 추가할 수 있나요?"
```

## 4. 검색(온라인) 전략

| 검색 | 방식 | 필터 | 후처리 |
|---|---|---|---|
| FAQ 인터셉트 | 벡터 + 임계값 | 업체 인스턴스 | 임계값 이상만 즉답 |
| 문서 RAG | **Hybrid(벡터+BM25)** | `company_id` | **Rerank** → Top-N |
| Tool RAG | Hybrid | (공용) | Rerank → Top-K |
| 자동완성 | prefix(주력) + 벡터(보강) | `company_id` | 인기도 정렬 |

### 4.1 파라미터(설정값, 튜닝 대상)

| 파라미터 | 초기값(제안) | 비고 |
|---|---|---|
| FAQ 임계값 | 0.85 (업체별 조정) | 보수적 시작(§01 §4.3) |
| 문서 RAG Top-K(검색) | 20 | Rerank 전 |
| 문서 RAG Top-N(생성 투입) | 4~5 | Rerank 후 |
| Tool RAG Top-K | 5 | 후보 도구 수 |
| 청크 크기 | 도메인 맞춤 | 메뉴 항목 보존 우선 |

> 키워드 중요(메뉴명·"양념"·"반반" 등)해서 **Hybrid 권장**. 정확한 메뉴명 매칭은 BM25가 강함.

## 5. 재인덱싱 정책

| 트리거 | 범위 | 방식 |
|---|---|---|
| 메뉴/정책 변경 | incremental | 변경 문서만 재처리 |
| 신규 업체 온보딩 | full | 전체 인덱싱(§09) |
| 임베딩 모델 교체 | full | 전체 재임베딩 |

- 관리자 API `POST /v1/admin/index`로 트리거(§03 §3.6).
- 작업 상태는 `index_jobs`(§04 §4.2)에 기록.
- **갱신 누락 = 잘못된 안내** → 변경 시 재인덱싱 + 자동완성 풀 동기화 필수.

## 6. 품질 고려사항

- 메뉴 가격/시간 등 **수치 정확도**가 신뢰의 핵심 → 청킹에서 수치-항목 분리 방지.
- 표 파싱 실패 시 품질 급락 → 파서 검증 + 폴백.
- FAQ와 문서 RAG 중복 답변 일관성 유지.
