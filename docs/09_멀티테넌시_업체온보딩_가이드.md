# 멀티테넌시 / 업체 온보딩 가이드

> **대상:** 백엔드/운영팀
> **목적:** 멀티테넌트 격리 모델과 새 업체를 추가하는 표준 절차를 정의한다.
> **관련 문서:** [01_아키텍처_설계서.md](./01_아키텍처_설계서.md) (§6) · [04_DB설계서.md](./04_DB설계서.md) · [08_MCP_서버_명세서.md](./08_MCP_서버_명세서.md)
> **상태:** 초안 (v1)

---

## 1. 테넌트 모델 개요

- 채택: **A안 (업체별 FAQ 서버/DB 인스턴스 격리)** — 법적 격리 + 비용 귀속(§01 §6).
- 원칙: **코드는 단일 템플릿, 데이터·인스턴스는 격리, 연결은 설정으로.**

### 파일럿 테넌트

| company_id | 업체 | FAQ 서버(예시) | FAQ 벡터DB(예시) |
|---|---|---|---|
| `pizza` | 피자집 | `pizza-faq.internal` | `qdrant-pizza:6333` |
| `chinese` | 중국집 | `chinese-faq.internal` | `qdrant-chinese:6333` |
| `chicken` | 치킨집 | `chicken-faq.internal` | `qdrant-chicken:6333` |

## 2. 격리 수준 (저장소별)

| 데이터 | 격리 | 메커니즘 |
|---|---|---|
| FAQ | **인스턴스 격리** | 업체 서버 = 업체 전용 벡터DB만 연결 |
| 문서 RAG | 논리 격리 | `documents` 컬렉션 + `company_id` 필터 |
| 자동완성 질문 | 논리 격리 | `autocomplete_q` + `company_id` 필터 |
| 관계형/캐시 | 논리 격리 | `company_id` 컬럼 / key prefix |

> 모든 검색·조회·캐시에 `company_id` 강제(§04 §2). 누락 = 격리 위반.

## 3. 두 가지 라우팅 (혼동 금지)

| 무엇을 고르나 | 방법 | 비고 |
|---|---|---|
| 어느 업체 서버? | **결정적**(레지스트리 조회) | company_id로 확정, 벡터검색 ❌ |
| 그 안에서 어느 FAQ/문서? | 의미기반 벡터검색 | 표현 달라도 매칭 |

> 업체를 벡터로 고르면 안 됨(오격리·법적 위반).

## 4. 테넌트 레지스트리

테넌트→서버/DB/설정 매핑. **하드코딩 금지**(§01 §4.2).

```yaml
# configs/tenants.yaml (예시)
tenants:
  pizza:
    name: "피자집"
    faq_server: "https://pizza-faq.internal"
    faq_vector_db: "qdrant-pizza:6333"
    faq_threshold: 0.85
    documents_collection: "documents"
    persona: "피자집 친절한 직원. 토핑·세트 추천에 능숙."
  chinese:
    name: "중국집"
    faq_server: "https://chinese-faq.internal"
    faq_vector_db: "qdrant-chinese:6333"
    faq_threshold: 0.88
    persona: "중국집 직원. 점심특선·단체주문 안내에 능숙."
  chicken:
    name: "치킨집"
    faq_server: "https://chicken-faq.internal"
    faq_vector_db: "qdrant-chicken:6333"
    faq_threshold: 0.85
    persona: "치킨집 직원. 후라이드/양념·부분육 옵션 안내."
```

> 레지스트리 단일 소스(yaml vs `companies` 테이블)는 결정 필요(§04 §7).

## 5. 새 업체 온보딩 절차 (체크리스트)

새 업체 추가 시 **코드 수정 없이** 아래만 수행한다.

1. **식별자 발급** — `company_id` 결정(예: `sushi`)
2. **전용 벡터DB 준비** — 업체 FAQ용 인스턴스/컬렉션 생성
3. **FAQ 서버 인스턴스 배포** — 템플릿 + config(company_id/db/threshold)
4. **레지스트리 등록** — `tenants.yaml`(또는 `companies`)에 한 줄 추가
5. **콘텐츠 수집** — 담당자에게 메뉴판/정책/FAQ 수령
6. **FAQ 적재** — `POST /v1/admin/faq` → 임베딩(§03 §3.6)
7. **문서 인덱싱** — `POST /v1/admin/index` (full) → documents + 자동완성 풀
8. **페르소나 설정** — 업체 톤/영업정보 등록
9. **검증** — 격리 테스트(요청 company_id 업체 데이터만 반환), 핵심 질문 스모크 테스트
10. **고객 채널 연결** — 업체 전용 위젯/링크에 `company_id` 주입(비회원, 고객이 직접 입력하지 않음)
11. **관리자 계정** — 담당자용 관리자 인증 발급(고객은 비회원이라 계정 없음)

## 6. 격리 검증 (필수 테스트)

| 테스트 | 기대 |
|---|---|
| `company_id=pizza`로 질의 | 피자집 데이터만 반환(중국집 데이터 미혼입) |
| 자동완성 후보 | 요청 company_id 업체 질문만 추천 |
| 세션/메시지 조회(다른 company_id) | `404`(노출 최소화) |
| 무효 company_id 전달 | `400 INVALID_REQUEST` |
| FAQ 적재(관리자) | 토큰 company_id 외 적재 불가 |

> 고객용은 비회원이라 "접근 통제"가 아니라 **"데이터 혼입 방지"**(요청 업체 데이터만 반환)를 검증한다. 관리자 적재는 인증 기반 격리 검증.

## 7. 비용 귀속

- 업체별 리소스(서버/DB/LLM 호출)에 **company_id 태깅** → 분리 정산(§01 §6).
- idle 업체 서버 비용은 스케일-투-제로 고려(결정 필요).
