# 멀티테넌시 / 업체 온보딩 가이드 (as-built)

> **대상:** 백엔드/운영팀
> **목적:** 멀티테넌트 격리 모델과 새 업체를 추가하는 **실제(as-built) 절차**를 정의한다.
> **관련 문서:** [12_구현_아키텍처.md](./12_구현_아키텍처.md) (구현 기준) · [08_MCP_서버_명세서.md](./08_MCP_서버_명세서.md) · [04_DB설계서.md](./04_DB설계서.md)
> **상태:** 구현 반영(v2)

## 1. 테넌트 모델 개요 (구현 기준)

- **MCP 서버 = 업체별 외부 독립 프로젝트.** 각 업체는 FAQ 서버 + 일반(도메인) 서버 2개를
  **독립 FastAPI+FastMCP 프로젝트**(자체 pyproject/venv/코드)로 보유한다 — 외부 벤더가 개발한 것처럼.
- **데이터 격리는 컬렉션/필터 + 벤더 적재.** 벡터DB(Qdrant)는 공유하되, FAQ 는 업체별 컬렉션
  (`faq_<id>`)으로, 문서/도구/자동완성은 `company_id` 필터로 격리. 데이터 적재는 각 벤더 서버가 소유.
- **연결은 설정으로.** 오케스트레이터는 `configs/tenants.yaml` 로 업체→서버 URL 을 결정적 라우팅.
  코드는 업체 비종속(하드코딩 없음).

> 설계 초안의 "업체별 전용 벡터DB 인스턴스(A안)"는 **컬렉션 분리로 확정**(인스턴스 분리는 운영 전환 시). [12 §5](./12_구현_아키텍처.md) 참고.

### 파일럿 테넌트

| company_id | 업체 | FAQ 서버 | 일반 서버 |
|---|---|---|---|
| `pizza` | 마르게리따 피자 | `http://127.0.0.1:9001/mcp` | `http://127.0.0.1:9101/mcp` |
| `chinese` | 행복반점 | `http://127.0.0.1:9002/mcp` | `http://127.0.0.1:9102/mcp` |
| `chicken` | 바삭치킨 | `http://127.0.0.1:9003/mcp` | `http://127.0.0.1:9103/mcp` |

## 2. 격리 수준 (저장소별, as-built)

| 데이터 | 격리 | 메커니즘 |
|---|---|---|
| FAQ | **컬렉션 분리** | `faq_<id>` 컬렉션 + 업체 faq 서버가 적재/조회 |
| 문서 RAG | 논리 격리 | `documents` 단일 컬렉션 + `company_id` 필터(+payload 인덱스) |
| 도구(Tool RAG) | 논리 격리 | `tools` + `company_id` 태깅(업체별 도구) |
| 자동완성 질문 | 논리 격리 | `autocomplete_q` + `company_id` 필터 |
| 관계형/캐시 | 논리 격리 | `company_id` 컬럼 / key prefix |

> 모든 검색·조회·캐시에 `company_id` 강제(§04 §2). **임베딩 규약(HashEmbedder·dim 384)은 오케스트레이터와 모든 벤더 서버가 동일해야** 공유 컬렉션이 맞물린다.

## 3. 두 가지 라우팅 (혼동 금지)

| 무엇을 고르나 | 방법 | 비고 |
|---|---|---|
| 어느 업체 서버? | **결정적**(tenants.yaml 조회) | company_id 로 확정, 벡터검색 ❌ |
| 그 안에서 어느 FAQ/도구? | 의미기반 벡터검색(FAQ 인터셉트 / Tool RAG) | 표현 달라도 매칭 |

> 업체를 벡터로 고르면 안 됨(오격리·법적 위반). 단계 1(라우팅 chitchat/rag/agent)은 기본 **규칙 기반**(`ROUTE_MODE=rule`).

## 4. 테넌트 레지스트리 — `configs/tenants.yaml` (실제 스키마)

```yaml
companies:
  bunsik:
    name: "김밥천국"
    status: active
    general_server_url: "http://127.0.0.1:9104/mcp"   # 벤더 일반(도메인) 서버
    persona: "분식집 직원입니다. 김밥·라면·떡볶이 안내에 능숙하고 친근한 말투."
    business_info:
      phone: "02-444-4444"
      address: "서울시 ..."
      hours: "매일 09:00~21:00"
      delivery: "..."
    faq:
      collection: faq_bunsik          # seed 데이터 적재용(임계값은 faq 서버가 소유)
      vector_db: "localhost:6333"
      server_url: "http://127.0.0.1:9004/mcp"   # 벤더 FAQ 서버
```

> 단일 소스(SSOT) = `tenants.yaml` 로 확정. `companies` 테이블은 운영 전환 시 동기화.
> `faq.threshold` 는 **오케스트레이터에 두지 않는다** — faq 벤더 서버 config(`FAQ_THRESHOLD`)가 소유.
> `tenants.yaml` 은 `@lru_cache` 로 기동 시 로드 → **수정 후 앱 재시작 필요.**

## 5. 새 업체 온보딩 절차 (체크리스트, as-built)

오케스트레이터 **코드 수정 없이**(= `tenants.yaml` 한 블록 + 앱 재시작) 아래를 수행한다.

1. **식별자 발급** — `company_id` 결정(예: `bunsik`).
2. **벤더 MCP 서버 2개 배포** (벤더 책임) — FAQ 서버(`match_faq`+`upsert_faq`) + 일반 서버
   (도메인 도구 + `ingest_documents`). 각자 독립 프로젝트/포트(예: 9004, 9104).
3. **레지스트리 등록** — `configs/tenants.yaml` 에 §4 블록 추가(persona/business_info/두 서버 URL).
4. **앱 재시작** — 레지스트리(lru_cache) 갱신 반영.
5. **데이터 적재** (벤더 소유):
   - 문서 → 벤더가 일반 서버 `ingest_documents(docs)` 호출 (또는 운영자가 데이터 전달).
   - FAQ → `POST /v1/admin/faq`(오케스트레이터가 faq 서버 `upsert_faq` 로 위임) 또는 벤더 직접.
6. **도구 디스커버리** — `python scripts/index_tools.py` 재실행 → 신규 일반 서버 `list_tools` 를
   `tools` 컬렉션에 적재(Tool RAG). `get_registry().all()` 을 돌므로 신규 업체 자동 포함.
7. **(선택) 자동완성** — 오케스트레이터가 문서로 질문 생성(`pipeline.generate_autocomplete`).
8. **검증** — §6 격리/스모크 테스트.
9. **고객 채널 연결** — 업체 전용 위젯/링크에 `company_id` 주입(비회원).
10. **관리자 계정** — 담당자용 관리자 토큰(`ADMIN_TOKEN`) (고객은 비회원).

## 6. 격리 검증 (필수 테스트)

| 테스트 | 기대 |
|---|---|
| `company_id=<신규>`로 질의 | 신규 업체 데이터만 반환(타 업체 미혼입) |
| 자동완성 후보 | 요청 company_id 업체 질문만 추천 |
| 세션/메시지 조회(다른 company_id) | `404`(노출 최소화) |
| 무효 company_id 전달 | `400 INVALID_REQUEST` |
| FAQ 적재(관리자) | 토큰 company_id 외 적재 불가 |

## 7. 비용 귀속

- 업체별 리소스(서버/LLM 호출)에 **company_id 태깅** → 분리 정산(§01 §6).
- idle 업체 서버 비용은 스케일-투-제로 고려(결정 필요).
