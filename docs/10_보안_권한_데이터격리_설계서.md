# 보안 / 권한 / 데이터 격리 설계서

> **대상:** 백엔드/운영/보안 담당
> **목적:** 인증·권한·테넌트 격리·개인정보 보호 기준을 정의한다.
> **관련 문서:** [01_아키텍처_설계서.md](./01_아키텍처_설계서.md) · [03_API명세서.md](./03_API명세서.md) · [04_DB설계서.md](./04_DB설계서.md) · [09_멀티테넌시_업체온보딩_가이드.md](./09_멀티테넌시_업체온보딩_가이드.md)
> **상태:** 초안 (v1)

> ⚠️ **구현 현황(as-built):** 격리는 **컬렉션 분리(FAQ) + `company_id` 필터(문서/도구/자동완성)**로 구현(인스턴스 분리 아님, [12](./12_구현_아키텍처.md) §5). 관리자 인증은 **단일 공유 토큰(`ADMIN_TOKEN`)** (per-company 토큰은 미구현). 데이터 적재는 업체별 외부 MCP 서버가 소유.
> - **업체 백엔드 DB 경계(추가)**: 정형/실시간 데이터는 업체 백엔드의 별도 DB(예: `pizza_shop`)가 소유. **오케스트레이터/MCP 는 이 DB 에 직접 접근하지 않고 백엔드 API 경유**(DB 접속정보가 오케스트레이터에 없음 → 추가 격리 경계, §01 §2.2).

## 1. 인증 모델 개요 (비회원)

- **고객용 API는 비회원(로그인 없음).** 어느 업체를 묻는지 구분하기 위해 **`company_id`를 요청에서 직접 받는다**(고객이 입력하는 게 아니라, 각 업체 전용 위젯/링크에 박혀 클라이언트가 자동 전달).
- **관리자 API는 인증 필수**(관리자 키/토큰). 공개 금지.
- 고객용 데이터(메뉴·영업·배달)는 **공개 정보**라 접근 통제를 두지 않는다. 대신 저장소 격리와 관리자 인증으로 핵심 리스크를 막는다.

## 2. 위협 모델 (요약)

| 위협 | 영향 | 대응 |
|---|---|---|
| 임의 company_id로 타 업체 **공개정보** 조회 | 낮음(공개정보) | 허용(의도된 설계). 저장소 격리는 유지 |
| 타 업체 **비공개/관리 데이터** 변조 | **법적/신뢰 치명** | 관리자 API 인증 + 역할 권한 |
| company_id 위조로 잘못된 응답 | 오안내 | 유효 company_id 검증(레지스트리 대조), 무효 시 `400` |
| 프롬프트 인젝션 | 시스템 지시 우회 | 입력=데이터 취급, 시스템 우선 |
| 개인정보 노출(주소/전화) | 법적 | 수집 최소화, 마스킹, 보존정책 |
| 관리자 API 오남용 | 데이터 변조 | 인증 + 역할 기반 권한 |
| 비회원 남용(스팸/비용폭증) | 운영비↑ | IP/세션 레이트리밋(§03 §4) |
| LLM 유해/범위밖 응답 | 신뢰↓ | 가드레일(§06) |

## 3. 인증 / 권한 (Authentication & Authorization)

| 구분 | 인증 | company_id | 권한 |
|---|---|---|---|
| 고객용(chat/autocomplete/sessions) | ❌ 비회원 | 요청에서 직접 수신 | 공개 |
| 관리자용(admin/*) | ✅ 관리자 키/토큰 | 토큰에서 취득 | 본인 업체 한정 |

- **수신한 company_id는 레지스트리와 대조해 유효성 검증.** 무효면 `400 INVALID_REQUEST`.
- 관리자 API는 적재/인덱싱 대상 테넌트를 **토큰의 company_id로 한정**(타 업체 변조 불가).
- 관리자 인증 방식(키 vs JWT vs IdP)은 결정 필요(§9).

## 4. 테넌트 데이터 격리 (핵심)

방어선 다중화(Defense in depth):

| 계층 | 격리 수단 |
|---|---|
| 인프라(벡터/메타) | FAQ는 **컬렉션 분리**(`faq_<id>`), 그 외 단일 컬렉션+`company_id` 필터(as-built) |
| **업체 백엔드 DB** | **벤더 소유 별도 DB**(예 `pizza_shop`). 오케스트레이터/MCP **직접접근 불가 — API 경유만**, 접속정보는 벤더 인프라에 격리 |
| 애플리케이션 | 레포지토리 계층에서 **수신 company_id**로 필터 항상 적용 |
| 캐시 | key prefix에 `company_id` |
| 라우팅 | 업체 선택은 결정적(수신 company_id → 레지스트리), 벡터검색 금지 |

- **검색 호출에 company_id 누락 = 버그(차단 대상).** 공통 의존성에서 주입, 누락 시 예외.
- 고객용은 접근 통제가 없으므로(공개정보), **격리의 의미는 "데이터가 섞이지 않음"**(요청한 업체 데이터만 반환)으로 한정된다. 비공개 데이터는 고객용에 두지 않는다.
- 정기 **격리 회귀 테스트**(§11 §5) CI에 포함 권장.

## 5. 개인정보 보호

음식 주문 도메인은 **주소·전화번호** 등 개인정보가 대화에 등장할 수 있다(배달 문의).

| 항목 | 정책 |
|---|---|
| 수집 최소화 | 파일럿은 주문 트랜잭션 미처리 → 개인정보 저장 회피 |
| 로깅 마스킹 | `query_logs`/`messages` 저장 시 주소·전화 패턴 마스킹 |
| 보존기간 | 메시지/로그 보존기간 설정 후 자동 파기 |
| 접근 통제 | 운영 조회도 company_id·역할 제한 |
| 응답 가드 | 개인정보 수집/노출 요청 거절(§06 가드레일) |

> `messages.content` 보존정책·마스킹 규칙은 결정 필요(§04 §7).

## 6. LLM 보안

| 항목 | 처리 |
|---|---|
| 프롬프트 인젝션 | 사용자 입력은 데이터로만, 시스템 지시 우선·불변 |
| 컨텍스트 격리 | 검색 컨텍스트에 타 업체 데이터 혼입 방지(필터) |
| 출력 검증 | verify 노드로 근거 외 답변 차단 |
| 비용/남용 | 레이트리밋(§03 §4), 입력 길이 제한 |

## 7. 시크릿 관리

- API 키(Anthropic 등)·DB 접속정보는 `.env`/시크릿 매니저. 코드/깃 커밋 금지.
- `.env`는 `.gitignore`, `.env.example`만 커밋.
