## 0. 두 축 — 테스트 vs 평가

| | 테스트(Test) | 평가(Evaluation) |
|---|---|---|
| 질문 | "동작이 맞는가?" (참/거짓) | "품질이 충분한가?" (점수) |
| 대상 | 코드·루프·격리·계약 | 검색 정확도·답변 품질 |
| 결정성 | 결정적(통과/실패) | 비결정적(지표·임계) |
| 도구 | pytest, 테스트 대역 | 평가셋 + 채점 |

> LLM/RAG는 출력이 비결정적이라 단순 `assert 정답==출력`이 불가. **테스트(루프·격리)**와 **평가(품질 점수)**를 분리해 다룬다.

---

## 1. 테스트 레벨

| 레벨 | 대상 | 예시 | LLM 필요 |
|---|---|---|---|
| 단위(Unit) | 도구 함수, 유틸 | `check_delivery_area` 입력→`ok/fail` | ❌ |
| 통합(Integration) | MCP 서버, 검색, 라우팅 | Tool RAG가 올바른 도구 Top-K 반환 | 일부 |
| E2E | 전체 대화 흐름 | "영업시간?" → FAQ 즉답 경로 | 대역 권장 |

피라미드: **단위 多 → 통합 中 → E2E 少**. 비용·속도 때문에 LLM 실호출 E2E는 최소화.

## 2. LLM 없이 테스트 (테스트 대역) ⭐

LLM API를 실제로 부르지 않고 **오케스트레이션 루프·분기·도구 호출**을 검증한다. (samplepkg `ScriptedChat` 패턴 차용)

### 2.1 모델 대역(stub)
"모델이 반환했을 법한 응답(도구 호출 포함)"을 **미리 스크립트로 재생**하는 가짜 LLM.

```python
# 예: 1턴째 check_delivery_area 호출, 2턴째 최종 답변
scripted = ScriptedChat([
    ToolCall("check_delivery_area", {"company_id": "chicken", "address": "..."}),
    FinalText("네, 해당 지역은 배달 가능하며 배달비는 3,000원이에요."),
])
result = orchestrator.run("여기 배달돼요?", llm=scripted, company_id="chicken")
assert result.route == "rag"
assert ("check_delivery_area", ...) in result.tool_calls
```

### 2.2 검증 포인트 (LLM 불필요)
- 도구 호출 루프가 결과를 올바르게 반영하는가
- 다단계/병렬 tool_calls 순회 처리(§06 §9)
- `max_iters` 초과 시 안전 종료
- FAQ 임계값 분기(이상→즉답 / 미만→통과)
- 답변 실패 폴백 분기(§06 §10)
- 타임아웃 처리(가짜 지연 주입 → 504/폴백, §03 §1.5.1)

> CI에서 빠르고 무료로 매번 실행. **운영 코드엔 대역 미포함.**

## 3. RAG 품질 평가

답변은 두 단계로 나눠 평가한다: **검색(Retrieval)**과 **생성(Generation)**.

### 3.1 검색 평가
"필요한 문서/FAQ가 후보에 들어왔는가."

| 지표 | 의미 |
|---|---|
| Recall@K | 정답 문서가 Top-K에 포함된 비율 |
| Precision@K | Top-K 중 관련 문서 비율 |
| MRR | 정답이 상위에 있을수록 높음 |
| Hit Rate | 정답 1개 이상 포함된 질의 비율 |

> Hybrid·Rerank 적용 전후 비교로 효과 검증(§07).

### 3.2 생성 평가
"답변이 근거에 충실하고 정확한가." (LLM-as-judge 또는 사람 채점)

| 지표 | 의미 |
|---|---|
| Groundedness(근거 충실성) | 답이 컨텍스트에 근거하는가(환각 여부) |
| Answer Relevance | 질문에 적절히 답했는가 |
| Correctness | 골든 답변과 사실 일치(가격·시간 등 수치 정확) |
| Completeness | 빠진 정보 없는가 |

> 음식점 도메인은 **가격·영업시간·배달비 수치 정확도**가 핵심 → Correctness 가중.

## 4. FAQ 임계값 튜닝

FAQ 즉답 임계값(§01 §4.3, §07)을 **데이터로** 결정한다.

| 방법 | 내용 |
|---|---|
| 평가셋 | FAQ에 해당하는 질문(다양한 표현) + FAQ 아닌 질문 혼합 |
| 측정 | 임계값별 **즉답 정답률 / 오답 즉답률 / 놓침률** |
| 목표 | **오답 즉답 최소화 우선**(보수적), 놓침은 RAG가 흡수 |
| 산출 | 업체별 최적 임계값(예: 피자 0.85 / 중국 0.88 / 치킨 0.85) |

```
질문 "몇 시까지 해요?" (= "영업시간?" FAQ)
  임계값 0.90 → 놓침(통과) → RAG가 답함(안전)
  임계값 0.75 → 즉답 but "주차돼요?"도 오즉답 위험
→ 오답 즉답이 없는 가장 낮은 값 탐색
```

## 5. 테넌트 격리 회귀 테스트 ⭐ (필수)

업체 데이터 혼입은 **법적 리스크**(§10). 매 배포 자동 검증한다.

| 테스트 | 기대 |
|---|---|
| 피자집 토큰으로 중국집 메뉴 질의 | 중국집 데이터 **미노출** |
| 자동완성 후보 | 자기 업체 질문만 |
| 세션/메시지 조회(타 업체) | `403 TENANT_FORBIDDEN` |
| FAQ 적재(타 업체) | 거부 |
| 검색 호출 company_id 누락 | 예외(차단) |

```python
def test_isolation_menu():
    res = chat("점심특선 뭐 있어요?", company_id="pizza")  # 중국집 메뉴 질문
    assert "짜장" not in res.answer and "탕수육" not in res.answer
    assert all(s.company_id == "pizza" for s in res.sources)
```

> 3개 업체 교차(피자↔중국↔치킨)로 매트릭스 구성. **CI 필수 게이트**.

## 6. 회귀 평가 (프롬프트/모델 변경)

프롬프트·모델·검색 파라미터 변경 시 품질 하락을 잡는다.

| 트리거 | 조치 |
|---|---|
| 프롬프트 수정 | 평가셋 재실행 → 이전 버전 대비 점수 비교 |
| 모델 교체(Haiku↔Sonnet 등) | 동일 평가셋으로 품질·비용·지연 비교 |
| 검색 파라미터(Top-K/임계값) | 검색 지표 재측정 |

> 임계값: 핵심 지표가 기준치 이하로 떨어지면 **배포 차단**.

## 7. 평가 데이터셋 (골든셋)

업체별로 **질문 → 기대 답변/기대 출처**를 구축한다.

```jsonc
{
  "company_id": "chicken",
  "question": "반반 시키면 양념 안 매운맛 돼요?",
  "expected_route": "rag",
  "expected_sources": ["menu_options"],
  "expected_answer_contains": ["반반", "양념", "맛 선택"],
  "category": "menu"
}
```

| 구성 | 내용 |
|---|---|
| 규모 | 업체당 수십~수백 케이스(카테고리별) |
| 카테고리 | 메뉴/영업시간/배달/주문/FAQ/실패유도(범위밖) |
| 출처 | 담당자 검수 + 실제 `query_logs` 표본 |
| 갱신 | 미응답 로그(matched=false)에서 신규 케이스 보충(§06 §10.4) |

## 8. 지표 ↔ KPI 연결 (§05)

| KPI(§05) | 측정 방법 |
|---|---|
| FAQ 즉답 적중률 70%+ | `query_logs` route=faq_intercept 비율 |
| TTFT 1.5초 이내 | 스트리밍 첫 토큰 지연 로깅 |
| 미응답 10% 이하 | `matched=false` 비율 |
| 정확도(피드백 up) 85%+ | `message_feedback` 집계 + 골든셋 Correctness |
| 건당 LLM 비용 | route별 토큰 사용량(`messages.usage`) 집계 |
