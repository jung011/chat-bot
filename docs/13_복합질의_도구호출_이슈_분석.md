# 13. 복합 질의 · 도구 호출 신뢰도 이슈 분석

> 복합 질의(예: "영업시간이랑 고르곤졸라 피자 얼마예요?")에서 agent 가 도구(`search_menu`)를
> 호출하지 않고 일부만 답하던 문제의 원인 규명 및 통제 실험 기록.
> 결론: **신뢰도 병목은 아키텍처가 아니라 LLM 도구 호출 방식(프롬프트-JSON vs 네이티브 함수호출)이었다.**

## 1. 증상

복합 FAQ+도구 질의에서 모델이 가격 도구를 회피하고 "도구 접근 불가"라며 일부만 답함.

| 질의 | 기대 | 실제(초기) |
|---|---|---|
| "고르곤졸라 피자 얼마예요?" (단독) | search_menu 호출 → 28,000원 | ✅ 정상 |
| "영업시간이랑 고르곤졸라 피자 얼마예요?" (복합) | 영업시간 + 가격 둘 다 | ❌ 영업시간만, "가격 도구 연결 안 됨" |

## 2. 복합 질의 판단 로직 (현재)

`app/orchestration/nodes/faq.py` 의 `_looks_compound()` — **순수 키워드/문장부호 휴리스틱**(LLM·임베딩·의미추론 없음):

```python
_COMPOUND_MARKERS = ("그리고","둘 다","되고","있고","이랑","랑 ","하고"," 및 "," 와 "," 과 ")
def _looks_compound(q):
    if (q.count("?") + q.count("？")) > 1: return True      # 물음표 2개↑
    return any(m in q for m in _COMPOUND_MARKERS)            # 연결어 부분문자열
```

- **복합으로 판정되면 0단계(FAQ 즉답)를 건너뜀** → rag/agent 가 FAQ·문서·도구를 병합(문서 §01 §2.2).
- 한계(부분문자열 매칭이라 취약):
  - **오탐**(단일→복합): "환불 **되고** 나서…", "맛**있고** 싼…" → 즉답 건너뛰고 rag/general 로(해는 없고 약간 느림).
  - **누락**(복합→단일): 연결어 없는 복합("영업시간 배달지역")은 못 잡음. 단, 복합 문자열은 임베딩이 희석돼 FAQ 점수가 낮아(threshold 0.78 미만) 대개 즉답으로 단락되지 않고 일반 경로로 흘러감 → 실전 피해는 제한적.

## 3. 근본 원인: claude_cli 의 프롬프트-JSON 이분법

`app/llm/cli_client.py` 의 `_tool_block` — claude_cli 는 **네이티브 도구 호출 미지원**이라
프롬프트로 JSON 출력을 유도한다:

```
도구가 필요하면: {"tool_calls":[...]}
답변 가능하면:   {"answer":"..."}
```

이 **tool_calls XOR answer 단발 형식**이 복합 질의의 "일부는 FAQ 로 답 가능 + 일부는 도구 필요"
상황을 표현하지 못한다. FAQ 컨텍스트가 주입되면 모델이 "이미 답할 게 있으니" `answer` 로
조기 종료하고 가격 도구를 건너뛴다(실측: 단독은 tool_calls, 복합은 answer 로 회피).

- **Tool RAG 검색 문제는 아님**: `search_menu` 는 복합 질의에서도 후보에 포함됨(다만 순위 하락:
  단독 1위 0.4 → 복합 2위 0.333, `get_business_hours` 가 1위). 즉 도구는 제공됐는데 모델이 안 부름.

## 4. 시도한 개선

1. **프롬프트 강화** (`_tool_block` + `AGENT_SYSTEM`): "질문 일부라도 도구로 확인 가능하면 먼저
   tool_calls, 목록의 도구를 '접근 불가'라 하지 마라" → 일관 실패(0/2) → **~75%(소표본)** 로 개선.
   하지만 비네이티브 한계로 0%↛100% 는 불가.
2. **아키텍처 변경(프로토타입)**: LLM 을 오케스트레이터 → general 서버로 이관(`general-pizza/app/agent.py`,
   `/answer`). 오케스트레이터는 FAQ 즉답 + 위임만 담당(`chat_service._delegate_and_answer`).

## 5. 통제 실험 — 구조 vs LLM API

동일 복합 질의 "영업시간이랑 고르곤졸라 피자 얼마예요?" **각 20회** 측정.
판정: 두 도구(`get_business_hours`+`search_menu`) 호출 + 가격(28,000)·영업시간 모두 답변.

| | claude_cli | OpenAI gpt-4o |
|---|---|---|
| **신 구조** (LLM=general 서버) | **9/20 (45%)**, 평균 42.2s, 타임아웃 1 | **20/20 (100%)**, 평균 3.5s |
| **구 구조** (LLM=오케+원격 MCP) | ~0~45%(초기 베이스라인) | **20/20 (100%)**, 평균 5.7s |

### 변수 분리

- **구조 고정·LLM 교체**(신 구조): 45% → 100% → **API 효과 +55%p (결정적)**
- **LLM 고정(OpenAI)·구조 교체**: 구 100% = 신 100% → **구조 효과 0%p (신뢰도 무관)**

## 6. 결론

- **신뢰도 병목은 LLM 도구 호출 방식이었다.** claude_cli 의 프롬프트-JSON 이분법(answer XOR
  tool_calls)이 복합 질의에서 도구를 건너뛰게 만들었고, **OpenAI 네이티브 함수 호출**(텍스트+tool_calls
  동시 + 결과 되먹임)로 바꾸자 100% 가 됐다.
- **아키텍처(general 서버에 LLM 이관)는 신뢰도와 무관**했다. 구 구조 그대로도 OpenAI 면 100%.
  구조 변경의 이점은 신뢰도가 아니라 **지연 ~2초 단축(원격 MCP 왕복 제거) + 벤더 자율성**.

## 6.1 전달 주장 대조 (인계 "복합 질의 모두 통과" vs 실측)

인계 시 **메신저로 "질의 케이스 모두 통과"** 를 전달받았고, **LLM 은 로컬 claude_cli** 를 썼다고 함.
그러나 우리 실측에선 복합 FAQ+도구가 45% 였다. 이 불일치는 **거짓이 아니라 검증 범위 차이**로 설명된다.

### 문서에는 "전 케이스 통과" 결과가 없다
docs 에 측정된 통과 결과는 없고 *설계 서술*만 있다. 유일하게 구체적인 복합 검증 케이스는:

> `docs/09 §6`: **"복합 질문(FAQ 2개 조합) | 두 FAQ 항목을 모두 답변(0단계 건너뛰고 rag 병합)"**

→ 딱 **FAQ+FAQ** 다. **FAQ+도구(가격) 조합은 문서의 검증 케이스에 없다**(`docs/08 §46` 에 "주차+마르게리따
가격" 이 *설계 의도*로만 언급).

### 복합 종류별 — claude_cli 기준

| 복합 종류 | 경로 | 도구 호출 결정 | claude_cli 결과 | 문서 검증 케이스 |
|---|---|---|---|---|
| FAQ+FAQ, FAQ+문서 | rag 병합 | 불필요 | ✅ 통과 | ✅ 있음(§09) |
| **FAQ+도구**(가격·주문 등) | agent | **필요** | ❌ **45%** | ❌ **없음(사각지대)** |

### 결론
- "모두 통과" 는 **rag-병합 복합(FAQ+FAQ/FAQ+문서) 범위 안에서는 사실**이었다(claude_cli 로도 통과).
- 실패한 **FAQ+도구(agent) 복합은 애초에 검증 케이스에 없던 사각지대**였고, claude_cli 이분법 한계로 45%.
- 즉 불일치의 원인은 LLM 백엔드 차이(가설①)가 아니라 **검증 범위(테스트한 복합 조합)의 차이**(가설②).

### 9.2 테스트 방법

- 질의 **2종 × 각 20회**를 `POST /v1/chat/sync` 로 전송(제보·§5와 동일 표본 크기).
- **성공 판정**: 두 의도(FAQ + 도구값)를 **모두** 답하고 **도구가 실제 호출**됨(`sources` 에 tool 포함). 도구를 건너뛰거나 값을 지어내면 실패.
- 호출 간 **2초 간격**(MCP 연결 부하 완화) + **매 실행 종료 후 4개 서버 health 재확인**(아래 9.5 함정 때문).

|---|---|:---:|:---:|
| "영업시간이랑 고르곤졸라 피자 얼마예요?" (FAQ+가격) | agent | **16/20 = 80%** | ~18.6s |

→ FAQ+도구 복합은 claude_cli 로 **~75~80%**. 100% 아님.

### 9.4 관측된 실패 양상 (서버 정상 상태에서)

| 양상 | 증거(원시 출력) |
|---|---|
| **XOR 형식 붕괴** | `{"tool_calls":[...]} {"answer":"..."}` 를 **동시 출력** → 파서가 도구 호출 인식 실패, 원본 JSON 그대로 노출 |
| **도구 미호출 + 값 환각** | `sources` 에 tool 없음(도구=[]) 인데 답변엔 "약 **35분**" 을 **지어냄** (DB 실측 30분과 다름) |
| **명시적 bail** | "저한테는 `estimate_delivery_time` 같은 도구가 실제로 연결되어 있지 않아서…", "도구가 현재 세션에서 직접 호출 가능한 상태가 아닌 것 같아요" |

> 근본 원인은 §3·§4 와 동일: claude_cli 는 네이티브 도구 호출이 없어 **`tool_calls` XOR `answer`** 단발 JSON 으로 흉내내는데, FAQ 컨텍스트가 주어진 복합 질의에서 형식이 무너지거나 도구를 건너뛴다. **값 환각(없는 35분 생성)** 은 특히 위험.

### 9.5 ⚠️ 측정 함정 — 서버 크래시 오염 (반드시 통제)

처음 두 번의 측정에서 같은 질의가 **0/20** 으로 나왔으나, 이는 LLM 이 아니라 **MCP 서버 크래시**였다.
- 20회 연속 호출의 streamable-http 부하에 **general-pizza / pizza-backend 가 `ConnectionResetError` 로 다운** → 도구 실행이 실제로 실패 → 모델이 "도구 연결 안됨"이라고 **사실대로 보고**(= LLM bail 처럼 보이지만 아님).
- 판별법: 실행 직후 `curl /health` 로 4개 생존 확인 + pizza-backend 로그에 `/delivery`·`/menu` 수신 기록 확인.
- 호출 간격(2초)을 주고 서버 생존을 확인하니 **0% → 75%** 로 정정됨.

> 즉 "0/20" 같은 극단값을 보면 **LLM 결론을 내리기 전에 서버 생존부터 확인**해야 한다. (Windows 에서 MCP 서버 연속 부하 안정성 자체가 별도 개선 과제)

### 9.6 제보("모두 통과")·외부 실패·우리 측정의 정합

- **rag-병합 복합**(FAQ+FAQ / FAQ+문서, 도구 불필요)은 claude_cli 로도 안정 통과 — 이것이 "모두 통과"의 실제 범위였다(§6.1, §09 §6 검증 케이스도 FAQ+FAQ).
- **FAQ+도구(agent) 복합**은 75~80% 비결정 → **single-shot 이면 자주 통과**(인계측 인상), **20회 반복하면 20~25% 실패 노출**(외부 사용자/우리 반복측정).
- 따라서 불일치는 거짓이 아니라 **표본 크기 + 검증한 복합 조합의 차이**(§6.1 가설②)이며, 본 측정이 이를 정량 확인. 단 실패율은 §5 의 45% 보다 낮은 **~20~25%**(프롬프트·agent 루프 개선 반영).

### 9.7 재현 절차 (step-by-step)

```bash

# 1) 서버 4개 (모두 fastembed, 데이터도 fastembed 시드 상태여야 함)
```
